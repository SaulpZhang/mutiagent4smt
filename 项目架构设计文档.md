# 项目架构设计文档

## 项目概述

本项目实现一个**多LLM智能体集成系统**，用于自动化IAM策略的形式化验证。系统通过多智能体协作生成SMT-LIB V2代码，实现云端IAM策略的自动化安全审计。

核心方法详见 [CLAUDE.md](CLAUDE.md)。

---

## 架构设计原则

1. **清晰** — 模块职责单一，命名直观，代码即文档
2. **可读** — 数据流透明，流程编排与业务逻辑分离
3. **可维护** — 低耦合高内聚，修改一个模块不影响其他模块
4. **可扩展** — 预留接口和扩展点，便于添加新功能或修改现有功能

## 架构分层详解

### 整体分层

```
┌────────────────────────────────────────────────────────┐
│                     pipeline/                           │
│   LangGraph StateGraph — 流程编排（唯一的"流"）           │
│   Node1 → Node2 → Node3 → ... (条件边实现回退循环)       │
└────────────────┬───────────────────────────────────────┘
                 │ 调用
┌────────────────▼───────────────────────────────────────┐
│                     modules/                            │
│   Input → Generation(A1+A2) → Evaluation(A3) → Output   │
│   纯业务逻辑，不感知 LangGraph，不感知 LLM 调用细节        │
└──────┬──────────────┬───────────────────┬──────────────┘
       │ 调用          │ 调用               │ 记录
┌──────▼──────┐ ┌──────▼───────┐  ┌─────────▼─────────┐
│   agent/    │ │   prompt/    │  │   experiment/      │
│  Agent工厂   │ │  Prompt管理   │  │  MetricsTracker    │
│  LLM客户端   │ │  静态模板文件  │  │  → SQLite          │
│  重试/超时   │ │  渲染/组合    │  │  ContextVar传播    │
└─────────────┘ └──────────────┘  └────────────────────┘
```

### 第一层：`core/` — 核心数据类型

**职责**：定义系统中所有模块之间通信的数据结构，作为模块间的"契约"。

**核心文件**：

- `core/schemas.py` — 所有 Pydantic v2 模型定义
- `core/exceptions.py` — 自定义异常类型

**设计理由**：
- Schema 即接口文档，新增模块只需定义新的输入/输出 schema
- 修改模块内部逻辑只要输入输出兼容，就不波及上下游
- Pydantic v2 提供自动序列化/反序列化，对实验记录和调试非常重要

### 第二层：`agent/` — Agent 基础设施层

**职责**：提供统一的 Agent 创建、管理和执行能力。将"如何调用 LLM"与"用 LLM 做什么"分离。

**核心文件**：

- `agent/base.py` — `BaseAgent` 抽象基类
- `agent/llm_client.py` — `LLMClient` 封装 API 调用（重试、超时、token 计数）
- `agent/factory.py` — `AgentFactory` 根据配置创建带特定 system prompt 的 Agent

**BaseAgent 设计**：

```python
class BaseAgent(ABC):
    llm_client: LLMClient
    system_prompt: str

    @abstractmethod
    async def run(self, context: dict) -> BaseModel:
        ...
```

**设计理由**：
- Agent 1/2/3 继承 `BaseAgent`，只需实现 `run()` 方法
- 切换模型（DeepSeek → GPT）只需改 `LLMClient`，业务代码不动
- 统一管理 API 调用、重试策略、超时控制

### 第三层：`prompt/` — 提示词管理

**职责**：管理所有静态 prompt 模板，提供加载和渲染接口。

**核心接口**：

```python
class PromptManager:
    def load(self, name: str, **variables) -> str:
        # 从 templates/ 读取模板文件，渲染变量
```

**设计理由**：
- Prompt 是静态文本，放在 `.txt` 文件中，非开发者也能修改
- 改 prompt = 改 `.txt` 文件，方便 git 版本对比
- 支持模板变量渲染，方便 prompt 策略实验

### 第四层：`modules/` — 业务逻辑模块

**职责**：实现 5 大模块（Input / Generation / Evaluation / Output / Verification）的具体业务逻辑。每个模块是一个独立的 Python 类，职责单一。

**模块列表**：

| 模块 | 类名 | 职责 |
|------|------|------|
| Input Module | `InputModule` | 解析验证指令和 IAM 配置，形成指令-配置对 |
| Generation Module | `GenerationModule` | 协调 Agent 1（意图理解→约束列表）和 Agent 2（代码生成+语法自修正） |
| Evaluation Module | `EvaluationModule` | Agent 3 评估约束满足情况，返回评估结果 |
| Output Module | `OutputModule` | 生成最终输出文件，包含代码和评估结果注释 |
| Verification Module | `VerificationModule` | Z3 求解器执行验证 |

**设计理由**：
- 完全解耦：模块不感知 LangGraph，不感知 LLM 调用细节
- 单元可测：传入 mock agent 即可单独测试模块逻辑
- 可替换：Agent 1 的 prompt 策略或整体算法可独立迭代

### 第五层：`pipeline/` — LangGraph 编排层

**职责**：作为系统的"流程唯一真理"，定义完整的执行图（节点 + 边 + 条件边）。

**核心文件**：

- `pipeline/state.py` — 图状态 `PipelineState` 定义
- `pipeline/nodes.py` — 节点适配器，将 `modules/` 包装为图节点
- `pipeline/graph.py` — `StateGraph` 定义，配置节点和边关系

**PipelineState**：

```python
class PipelineState(TypedDict):
    input: VerificationInput | None
    constraints: ConstraintsList | None
    code: SMTLibCode | None
    evaluation: EvaluationResult | None
    output: OutputResult | None
    iteration: int
    max_iterations: int
    tracking: ExperimentRecord  # MetricsTracker 绑定在此
```

**设计理由**：
- 流程图和数据流完全透明 — 打开 `graph.py` 就看清系统运作方式
- 条件边天然实现"不满足 → 回退 Agent 2"的迭代循环
- 要改变流程（新增节点、修改回退逻辑），只改这个层

### 第六层：`experiment/` — 实验追踪（横切关注点）

**职责**：记录每个实验用例的完整信息，包括输入、输出代码、执行结果、标签、耗时等。

**核心机制** — ContextVar + 上下文管理器：

```python
import contextvars
current_record: ContextVar[ExperimentRecord] = contextvars.ContextVar("current_record")

class MetricsTracker:
    @contextmanager
    def track_stage(self, stage_name: str):
        record = current_record.get()
        start = time.perf_counter()
        yield
        elapsed = time.perf_counter() - start
        record.stages[stage_name] = StageRecord(duration_ms=elapsed)
```

**每个实验用例记录的字段**：

| 字段 | 来源 |
|------|------|
| experiment_id | 自动生成 UUID |
| instruction | Input Module |
| account_data | Input Module |
| constraints_list | Agent 1 |
| generated_code | Agent 2 |
| code_execution_result | Z3 / syntax check |
| evaluation_result | Agent 3 |
| all_satisfied | Agent 3 |
| num_iterations | Pipeline 迭代计数 |
| stage_timings | 各阶段耗时 |
| label | answers.json 中的标签 |
| timestamp | 自动 |
| model_used | 配置 |

**设计理由**：
- `ContextVar` 避免了将 tracker 对象层层传递到每个函数签名
- 上下文管理器保证异常发生时已记录的数据不会丢失
- Pipeline 执行完毕后自动写入 SQLite
- SQLite schema 可扩展新字段，不影响历史数据

## 完整数据流

```
Input Module
  │  instruction + account_data
  ▼
Agent 1 (Intent Understanding)
  │  constraints_list (结构化约束列表)
  ▼
Agent 2 (Code Generation)
  │  smt_code
  ├── 语法检查失败 ──→ 自修正循环 (最多N次)
  └── 语法检查通过
      │
      ▼
Agent 3 (Evaluation)
  │  evaluation_result
  ├── all_satisfied = true  ──→ Output Module ──→ Verification Module (Z3)
  └── all_satisfied = false ──→ Agent 2 (iteration+1，回退重试)
                                  │
                                  └── iteration > max_iterations
                                      │
                                      ▼
                                  Output Module (输出当前最佳结果)
                                          │
                                          ▼
                                   ExperimentRecorder → SQLite
```

## 可扩展性

| 扩展需求 | 需要修改的位置 | 不影响的范围 |
|---------|--------------|-------------|
| 新增一个 Agent | `modules/` 新增模块，`pipeline/graph.py` 加节点 | 其他 Agent 的业务逻辑 |
| 更换 LLM 模型 | `config.py` 改 model name | 所有业务代码 |
| 修改生成 prompt | `prompt/templates/` 改对应 txt | 全部代码 |
| 调整迭代逻辑 | `pipeline/graph.py` 改条件边 | `modules/` 业务逻辑 |
| 新增评估指标 | `experiment/models.py` 加字段 | 已有指标和历史数据 |
| 切换 LLM 框架 | `agent/llm_client.py` | `modules/` 业务代码 |
| 添加新模块（如缓存模块） | `modules/` + `pipeline/graph.py` | 现有模块无需改动 |

## 依赖关系

```
pipeline/  ──依赖──▶  modules/  ──依赖──▶  agent/  ──依赖──▶  core/
                        │                      │
                        ▼                      ▼
                   prompt/                 core/schemas.py
                        │
                        ▼
                   experiment/  ──依赖──▶  core/schemas.py
                        │
                        ▼
                   SQLite (持久化)
```

**依赖方向**：高层次的层依赖低层次的层，低层次绝不依赖高层次。
